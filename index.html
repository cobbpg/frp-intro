A Short Primer

Gergely Patai

What’s in the name?

‘Functional’

  • Referentially transparent building blocks, i.e. inlining or extracting definitions doesn’t change the meaning of the program
  • Functions as the primary means of composition
  • Ability to perform equational reasoning

‘Reactive’

A program that interacts with its environment at a speed dictated by the environment. (Gérard Berry, 1989)

In other words, (soft or hard) real-time systems, e.g. embedded systems or games.

BUT: Nowadays the term ‘reactive programming’ seems to be strongly associated with data-flow networks or streams.

The original core idea

  • Treat the whole lifetime of a time-varying quantity as a value
  • Two basic abstractions and their denotations:
    • ⟦Behaviour〈α〉⟧ = Time → α
    • ⟦Event〈α〉⟧ = Stream〈(Time, α)〉
  • Examples:
    • microphone : Behaviour〈Float〉
    • keyboard : Event〈KeyCode〉

Examples

Set position on mouse click

ballPosition = hold(snapshot(mousePosition, mouseClick))

snapshot : (Behaviour〈α〉, Event〈β〉) → Event〈α〉

hold : Event〈α〉 → Behaviour〈α〉

Follow mouse on a spring

ballPosition =
integral(integral(k * (mousePosition - ballPosition)))

k = abs(lowPass(microphone)) 😁

What do we gain?

Lightweight
encapsulation

No need to introduce new variables when adding more stateful processing steps.

Complex event handling

No callback hell. The data-flow and temporal behaviour is plain to see.

Transactionality

Respecting referential transparency implies that we cannot observe glitches or inconsistent state.

Explicit state changes

No side-effects. State is only mutated by localised logic and the ordering of events is explicitly controlled.

No hidden dependencies

Everything that can affect the behaviour of an entity must be accounted for at the time of its declaration.

Testability

Thanks to everything being just a value, we can easily apply QuickCheck-style property tests to verify our logic.

Different approaches

  • What are the first-class building blocks?
  • How can we combine them?

Building blocks

  • Events and behaviours (reactive values, RVs) are first-class elements.
  • Transformers of RVs are first-class elements, while RVs themselves are only implied.
  • Both approaches can be equally expressive!

‘Value oriented’

  • We have types like Event〈α〉 and/or Behaviour〈α〉, which represent time-varying values.
  • We combine values of these types with plain functions, like snapshot or hold.
  • Preferred option for a textual language.

‘Transformer oriented’

  • We have types like Trans〈α, β〉, which represent transformations between time-varying quantities.
  • We combine these transformations with functions like:
    • ser : (Trans〈α, β〉, Trans〈β, γ〉) → Trans〈α, γ〉
    • par : (Trans〈α, β〉, Trans〈γ, δ〉) → Trans〈(α, γ), (β, δ)〉
  • Preferred option for a visual language.
  • Typically associated with Arrows.

Structure

  • Can be static, dynamic, or hybrid (only certain pre-defined forms of dynamism possible).
  • Trade-off between the potential extent of static analysis (including the ability to optimise) and expressiveness.

Dynamism

  • Support for higher-order reactive values or transformations.
  • switch : Event〈Behaviour〈α〉〉 → Behaviour〈α〉
  • switcher : Trans〈(Trans〈α, β〉, α), β〉

Final takeaway

  • FRP is a principled way to organise state and manage state changes.
  • There’s no one true way of doing FRP; different applications require different designs.
  • You can use these ideas in the small or large, or just to inform your design decisions.
  • The jury is still out on whether using FRP pervasively causes too much friction to be worth it.
